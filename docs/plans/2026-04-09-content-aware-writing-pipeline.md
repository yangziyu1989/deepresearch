## Problem Statement

Tao can run a full research pipeline (topic to paper) autonomously, but the output quality isn't publication-ready. The AI Scientist (Sakana, Nature 2026) showed that fully autonomous papers are "mediocre" even when they pass workshop review. The quality gaps — naive ideas, shallow experiments, hallucinated citations, AI-sounding prose — are exactly the ones humans catch in 30 seconds.

The opportunity: instead of competing on autonomy (AI Scientist, PaperOrchestra), compete on speed-with-quality. A researcher using Tao in A1 mode should compress 3 months of work into a weekend by having the system do 90% of the grunt work while surfacing the critical 10% of decisions to the human.

## What Makes This Cool

The "whoa" moment: an ML researcher opens Tao, types a research question, and within hours has a paper draft where they only made ~15-20 decisions (pick the narrative, approve the pivot, select the figure, judge the experiment quality). Everything else — lit search, coding, GPU experiments, LaTeX, figures, citations — happened autonomously. They did a weekend of focused creative work. The system did 3 months of execution.

## Constraints

- Must work within the existing Tao pipeline (20 stages, RunPod GPUs)
- A1 decisions use Claude Code's AskUserQuestion (single/multi choice)
- Decision log (logs/decisions.jsonl) captures everything for reproducibility
- A0 mode must still work (fully auto, best-scoring picks)
- Budget: RTX 4090 tier (~$0.34/hr), not A100s

## Premises

1. The quality gap in autonomous research papers comes from ~10% of decisions that require human taste/judgment — topic selection, experiment direction after failures, paper narrative framing, and figure aesthetic choices.
2. A1 mode (human-in-loop) is Tao's competitive advantage over AI Scientist and PaperOrchestra, which are fully autonomous.
3. The target user is an ML researcher who brings domain expertise and research taste. They don't want to write boilerplate or run experiments manually, but they want to steer the creative direction.

## Cross-Model Perspective

Landscape search revealed:
- AI Scientist v2 passed workshop review but creator called output "mediocre." 31% citation hallucination rate even after multiple checks.
- PaperOrchestra (Google, April 2026) focuses on writing quality with multi-agent pipeline and PaperWritingBench.
- Agent Laboratory found literature review is the highest-failure phase.
- The gap: no system optimizes for human-AI collaboration quality. All compete on autonomy.

### Approach B: Content-Aware Writing Pipeline
Redesign writing as multi-agent pipeline (PaperOrchestra-style) with domain-specific agents and quality scoring.
- Effort: L | Risk: Medium
- Pro: Addresses writing quality structurally

### Approach C: Human-in-Loop Advantage (CHOSEN)
Make A1 the killer mode. System does all grunt work, surfaces exactly the right decisions to the human at exactly the right time.
- Effort: M | Risk: Low
- Pro: Plays to Tao's unique strength, sidesteps "mediocre AI" problem
- Con: Less flashy than "fully autonomous"

### Critical Architecture: How decide() Works

The Tao pipeline is orchestrated by a Python `Lifecycle` class that generates `Action` objects. Claude Code executes these actions. The key insight: **`decide()` does not call `AskUserQuestion` directly.** Instead:

**A1 flow:**
1. `Lifecycle.get_next_action()` reaches a decision point
2. Instead of returning a normal `Action`, it returns a **`Decision` action** — an `Action` with `action_type="decision"` containing the question, options, and context
3. The Claude Code session receives this action and invokes `AskUserQuestion` (Claude Code tool)
4. The human responds via the Claude Code UI
5. Claude Code calls `cli_record()` with the chosen option as the result
6. The pipeline advances based on the recorded choice

This means `decide()` is NOT async Python. It's a **protocol**: the pipeline emits a decision action, yields control to Claude Code, and resumes when the result is recorded. No WebSocket, no HTTP bridge — same mechanism as every other action in the pipeline.

**A0 flow:**
1. Same decision point, but `Lifecycle` checks `config.autonomy_level == 0`
2. Instead of emitting a `Decision` action, it auto-selects the highest-scored option
3. Logs the auto-decision to `logs/decisions.jsonl`
4. Returns the next normal `Action` — Claude Code never sees the decision

**Option scoring for A0:** Each decision builder function assigns scores using:
- LLM-generated scores (for subjective decisions like narrative framing)
- Metric-based scores (for quantitative decisions like "acceptance rate > threshold?")
- Default ordering (first option = recommended, score 1.0; others descending)

**A1 timeout:** If no human response within 4 hours, the pipeline auto-falls back to A0 behavior for that single decision (picks best-scoring option) and logs `chosen_by: "auto_timeout"`. Pipeline continues. Human can review the auto-decision in `logs/decisions.jsonl`.

## Resolved Design Decisions

1. **Fast track (pre-approve decision types)?** — MVP: no. All A1 decisions pause. Add per-type override in v2 if users request it.

## Success Criteria

1. A researcher using A1 mode can go from topic to submittable paper draft in <48 hours of wall-clock time, making ~15-20 decisions
2. The paper quality (measured by self-review score) is ≥7/10, vs current ~5-6/10 for A0
3. Decision points feel natural, not interruptive — the system pauses at moments where the human genuinely adds value
4. All memories/learnings from this session are applied automatically in the next run
